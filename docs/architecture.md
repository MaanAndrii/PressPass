# Архітектура PressPass Platform

## Загальна схема

```
┌────────────────────────────┐
│        PWA (Next.js)       │  журналіст: /card
│  React + Tailwind + SW     │  адмін:     /admin
│                            │  публічно:  /verify/{uuid}
└─────────────┬──────────────┘
              │ HTTPS / REST (JSON, Bearer JWT)
┌─────────────▼──────────────┐
│       REST API (NestJS)    │  auth, me, verify,
│  Helmet · CORS · Throttler │  admin/journalists, admin/cards
└─────────────┬──────────────┘
              │ Prisma ORM
┌─────────────▼──────────────┐
│         PostgreSQL         │  users · journalists · cards
└────────────────────────────┘
```

Ключовий принцип: **фронтенд не має власної бізнес-логіки**. Уся логіка (статуси, генерація UUID/номерів, права доступу) виконується на сервері; PWA лише відображає відповіді REST API.

## Монорепозиторій

npm workspaces:

- `apps/api` — NestJS. Модулі: `auth`, `me`, `verify`, `journalists`, `cards`, `prisma` (глобальний).
- `apps/web` — Next.js App Router. Сторінки: `/login`, `/card`, `/verify/[uuid]`, `/admin/*`.
- `packages/shared` — типи контрактів API (`CardResponse`, `VerifyResponse`, …) та чисті функції (`buildVerifyUrl`, `effectiveCardStatus`). Використовується і бекендом, і фронтендом — контракт один.
- `packages/ui` — спільні React-компоненти (Button, Badge, Field).
- `packages/config` — спільні пресети tsconfig.

## Авторизація

- Вхід: email + пароль → `POST /auth/login` → JWT (підпис `JWT_SECRET`, строк `JWT_EXPIRES_IN`).
- Паролі: **Argon2id**.
- `JwtAuthGuard` — глобальний guard; публічні маршрути позначаються `@Public()` (`/auth/login`, `/verify/:uuid`).
- `RolesGuard` + `@Roles(...)` — доступ до `/admin/*`. JWT містить `role` та (для редакційного адміна) `editorialId`.
- **Два рівні адміністрування**: `ADMIN` — системний адміністратор (усе + керування адміністраторами); `EDITORIAL_ADMIN` — прив'язаний до `editorial_id`. Спільні ендпоінти дозволені обом ролям, а **обмеження за редакцією накладає сервіс**: редакційний адмін бачить/змінює лише картки й профіль своєї редакції, а `editorialId` при видачі підставляється з токена (не з тіла запиту). Створення/видалення редакцій і адміністраторів, дизайн і налаштування — лише `ADMIN`.
- Logout збільшує `tokenVersion`, відкликає JWT і знищує in-memory unlock sessions.

## Посвідчення та статуси

- `uuid` — **UUIDv7**, генерується сервером при видачі. Це єдиний ідентифікатор, що потрапляє в QR.
- У БД зберігаються статуси `ACTIVE` / `BLOCKED`; `EXPIRED` **обчислюється** з `expire_date` при читанні (`effectiveCardStatus`). Це усуває потребу у фонових задачах: посвідчення стає недійсним наступного дня після дати завершення автоматично.
- `BLOCKED` має пріоритет над датами; `renew` подовжує строк і повертає статус `ACTIVE`.
- **Редакція-емітент**: посвідчення можна видати від імені компанії з довідника `editorials` (розділ адмінки «Редакції»). Тоді назва й логотип редакції беруться з реєстру компаній і виводяться на картці та сторінці перевірки; звʼязок `cards.editorial_id` необовʼязковий (`ON DELETE SET NULL`), тож видалення редакції не руйнує історію виданих посвідчень.

## Дизайн посвідчення (дані, а не код)

- Вигляд картки задає JSON-шаблон (`card_templates`, singleton), який змінюється з адмінки (`/admin/design`) без перевидання посвідчень. Шаблон санітизується на сервері (білий список ключів, hex-кольори, числові межі, enum-и) — довільний HTML/скрипт неможливий.
- **Структура під екран смартфона**: картка має фіксовані пропорції (`cardWidth × cardHeight`) і рівномірно масштабується (CSS `transform: scale`, коефіцієнт рахує хук `useFitScale`), щоб уміститися на будь-якому екрані без зміни пропорцій. Хедер і футер — фіксованої висоти (`headerHeight`, `footerHeight`).
- **Три зони основної частини**: верхня (фото + ПІБ і посада), середня (логотип + редакція), нижня (номер + дата дії). Кожна зона позиціюється вільно — по горизонталі (зліва/центр/справа) і вертикалі (зверху/центр/знизу) — і має власний інтерліняж (`theme.zones.{top,middle,bottom}`).
- Поля (ПІБ, посада, редакція) показуються значеннями без підписів за замовчуванням (підписи вмикаються перемикачем). ПІБ — з профілю журналіста; **посада — з посвідчення** (її задає редакція при видачі); **редакція — дисплей-назва компанії-емітента** з реєстру. Типографіка налаштовується: сімейство шрифтів (web-safe стеки, без зовнішніх шрифтів), масштаб і інтерліняж. Дата видачі на картці не показується.
- **Двомовність**: головна сторона — повністю українська, свайп (або тап) робить 3D-переворот на повністю англомовну версію. Один компонент `CardTemplateView` рендериться двічі (`lang="uk"` / `lang="en"`); англійська сторона бере окремі поля (`full_name_en`, `position_en`, `organization_en`, англомовні заголовок/підпис) з фолбеком на українські значення.
- Екран `/card`: без кнопок — закриття хрестиком ✕ у куті, оновлення свайпом вниз (pull-to-refresh); QR автооновлюється кожні 30 с.

## QR (динамічний, захищений від підробок)

QR містить лише URL: `{VERIFY_BASE_URL}/verify/{uuid}?t={токен}` — жодних персональних даних.

Захист від підробок:

- **Короткоживучий підписаний токен** — HMAC-JWT з полем `exp` (`QR_TOKEN_TTL`, типово 60 с), згенерований тільки сервером і привʼязаний до UUID конкретного посвідчення. Стан ніде не зберігається — перевірка це підпис + строк дії.
- **Динамічне оновлення** — застосунок запитує `GET /card/qr` і перемальовує QR кожні 30 секунд. Скриншот, роздрук чи копія чужого QR перестає проходити перевірку за хвилину.
- **Сервер не розкриває даних без токена** — прострочений/відсутній/чужий токен → `{valid:false, qrStatus}` без жодних персональних даних; перебір UUID нічого не дає. Сторінка перевірки в цьому разі показує «QR-КОД НЕДІЙСНИЙ» з інструкцією попросити свіжий QR з екрана.
- Вся перевірка виконується сервером по HTTPS; фронтенд лише відображає результат.

Наслідок дизайну: для проходження перевірки телефон журналіста має бути онлайн (щоб отримати свіжий токен). В офлайні посвідчення видно, але QR може бути простроченим — застосунок попереджає про це.

## PWA / офлайн

- `manifest.webmanifest` + іконки → встановлення на головний екран (Android, iOS).
- Service worker (`public/sw.js`): app shell кешується при встановленні; всі успішні GET-відповіді (включно з `GET /card`) кешуються під час роботи. Без мережі показується останнє завантажене посвідчення (SRS §11).

## Фото

Файли приймаються в RAM, перевіряються за magic bytes і шифруються випадковим File DEK у `uploads/encrypted/`; File DEK wrapped owner key. Direct static access відсутній, читання виконується authorization-aware `/media/:id` або короткоживучою QR capability.

## Безпека

| Вимога        | Реалізація                                                      |
| ------------- | --------------------------------------------------------------- |
| HTTPS         | термінується на Nginx (production)                              |
| JWT           | `@nestjs/jwt`, глобальний guard                                 |
| Argon2        | хешування паролів                                               |
| UUIDv7        | пакет `uuid` (v7), генерація тільки на сервері                  |
| Rate limiting | `@nestjs/throttler`: 100/хв глобально, 5/хв login, 30/хв verify |
| CORS          | білий список із `CORS_ORIGIN`                                   |
| Helmet        | security-заголовки                                              |
| SQL Injection | Prisma (параметризовані запити), валідація DTO                  |
| XSS           | `ValidationPipe` (whitelist), React-екранування, Helmet         |

## Розширюваність (v2)

Сервер не змінюється при додаванні клієнтів: Flutter-застосунок, Google/Apple Wallet і push працюватимуть через той самий REST API (окремі модулі-адаптери додаються поруч, не змінюючи наявні). PostgreSQL, NestJS, REST API і структура БД залишаються незмінними.

## Деплой (ціль MVP)

```
Nginx (HTTPS, статика, проксі)
 ├── /            → Next.js (web)
 ├── /api/*       → NestJS (api)   (або окремий піддомен)
 └── /api/media/* → авторизоване розшифрування файлів
PostgreSQL — локальний інстанс
```
